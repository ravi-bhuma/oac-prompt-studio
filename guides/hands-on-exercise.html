🟢 Beginner Track — Core Agent Behaviors
1
Attribute Grouping
Use Case #1 · Technique: Context Injection
Behavior
Objective

Teach the agent to group raw Order Priority values into business-friendly categories: "Regular" and "Important".

Why this matters

Your dataset has values like Low, Medium, High, Critical — but your team says "Regular Orders" and "Important Orders". Without this instruction, the agent shows all raw values.

Add this to your Supplemental Instructions:
When user asks about Important Orders or Regular Orders: - "Low", "Medium", "Not Specified" → mapped to "Regular" Orders - "High", "Critical" → mapped to "Important" Orders TRIGGERS: "important", "regular", "important vs regular", "order priority" Always use the group name in chart labels, not raw values.
🧪 Test It — Ask Your Agent:

1. "Compare Revenue for Regular and Important Orders" → Should group and show 2 bars

2. "Show Average Profit for Important Orders" → Should filter to High + Critical only

3. "Revenue by Order Priority" → Should show grouped categories, not raw values

✅ Success Criteria

Agent consistently groups Low+Medium+Not Specified as "Regular" and High+Critical as "Important" in all responses.

2
Set a Default Metric
Use Case #3 · Technique: Constraints
Behavior
Objective

When users ask vague questions without naming a metric, the agent should default to Revenue.

When the user does not specify a metric, default to "Revenue" as the primary measure. Apply this default when the user says: - "Show performance across [dimension]" - "Compare [A] vs [B]" - "How are we doing?" Only override when the user explicitly names a different metric.
🧪 Test It — Ask Your Agent:

1. "Compare East vs West" → Should use Revenue (not Profit or Quantity)

2. "How are we doing this quarter?" → Should show Revenue

3. "Show profit by region" → Should use Profit (user specified it)

3
Persistent Default Filter
Use Case #4 · Technique: Constraints
Behavior
Objective

Lock in a mandatory City filter so every query is scoped to one location.

Always apply the following filter on every query: City = "Toronto" This filter is mandatory. Do not remove unless user explicitly says "remove the city filter" or "show all cities". Include in every response: "Filtered to: Toronto"
🧪 Test It — Ask Your Agent:

1. "Revenue by Product Category" → Should auto-filter to Toronto

2. "What about Vancouver?" → Should temporarily switch to Vancouver

3. "Show all cities" → Should remove filter and show everything

4
Default Visualization Types
Use Case #5 · Technique: Format Control
Behavior
Objective

Define automatic chart selection rules based on query type.

Apply the following default chart rules: - Comparison ("compare", "vs", "across") → Horizontal Bar Chart - Time-based dimension (Date, Month, Quarter, Year) → Line Chart - Geographic attributes (City, State, Country) → Map - Ranking (Top N / Bottom N) → Table with sorted values - Distribution or segmentation → Stacked Bar or Pie Chart Only override if user explicitly requests a different chart type.
🧪 Test It — Ask Your Agent:

1. "Compare Revenue across Product Categories" → Should show Horizontal Bar

2. "Monthly sales for 2025" → Should show Line Chart

3. "Revenue by City" → Should show Map

5
Top N / Bottom N Ranking
Use Case #6 · Technique: Few-Shot + Constraints
Behavior
Objective

Standardize ranking behavior with default count, metric, and keyword mapping.

When user asks for "Best", "Highest", "Top performing": → Rank by Revenue descending When user asks for "Bottom", "Least", "Worst": → Rank by Revenue ascending Default metric: Revenue. Default count: 5. Keyword mapping: - TOP: best, highest, most, leading, top, largest - BOTTOM: worst, least, lowest, bottom, smallest
🧪 Test It — Ask Your Agent:

1. "Show me the best products" → Top 5 by Revenue descending

2. "Bottom 3 cities by profit" → 3 cities, lowest Profit ascending

3. "Which customers are our largest?" → Top 5 by Revenue

🟡 Intermediate Track — Calculations & Time Logic
6
Aggregation Rules
Use Case #7 · Technique: Constraints
Calculations
Objective

Define correct aggregation for each measure — wrong aggregation = silently wrong numbers.

Default aggregation rules: - Revenue: Sum(Revenue) - Profit: Sum(Profit) - Quantity: Sum(Quantity) - Discount: Average(Discount) - Order Count: Count Distinct(Order_ID) For unlisted numeric fields, default to Sum. User keywords ("average", "count", "sum") override defaults.
🧪 Test It — Ask Your Agent:

1. "Discount by Product Category" → Should use Average, NOT Sum

2. "How many orders per region?" → Should use Count Distinct

3. "Average revenue by segment" → User said "average" — should override Sum default

7
Custom Formulas & KPIs
Use Case #8 · Technique: Context Injection
Calculations
Objective

Give the agent exact formulas for custom KPIs that aren't in the dataset.

Custom KPI formulas: KPI: Average Transaction Value (ATV) Formula: Sum(Sales_Revenue) / Count Distinct(Order_ID) TRIGGERS: ATV, average transaction value KPI: Profit Margin % Formula: (Sum(Profit) / Sum(Revenue)) * 100 TRIGGERS: margin, profit margin KPI: Sales per Sq. Ft Formula: Sum(Sales_Revenue) / Sum(Selling_Area_sqft) TRIGGERS: Sales per Sq. ft, Sales per Square feet For all KPIs: - If denominator is zero, return "N/A" - Round to 2 decimal places
🧪 Test It — Ask Your Agent:

1. "What is our profit margin by region?" → Should compute formula, show % values

2. "ATV per Product Category" → Should divide Revenue by distinct orders

3. "Margin for a category with zero revenue" → Should return N/A

8
Hierarchies & Drill-Down
Use Case #2 · Technique: Context Injection
Behavior
Objective

Define parent→child drill paths for smooth hierarchical navigation.

Enable hierarchical drill-down navigation: Product: Product Category > Product Sub-Category > Product Name Geography: Region > Country > State > City Drill-Down Logic: - Drill only when current chart contains a column from this hierarchy. - If showing Product Category, on "Drill Down" → show Sub-Category within selected. - If at Product Name level, inform: "This is the most detailed product level." - If column is not in any hierarchy: "Drill-down is not available for this attribute."
🧪 Test It — Ask Your Agent:

1. "Revenue by Product Category" → then "Drill Down on Technology" → Should show Sub-Categories

2. "Drill down on Discount column" → Should say "not available for this attribute"

9
Fiscal Calendar Mapping
Use Case #11 · Technique: Context Injection
Time
Objective

Map your non-standard fiscal year so time queries return correct results. Note: Update these hardcoded values when your fiscal periods change.

Use the Fiscal Calendar (FY = Jun 1 – May 31). Quarters: Q1 Jun–Aug, Q2 Sep–Nov, Q3 Dec–Feb, Q4 Mar–May. Columns (exact, with spaces): "Fiscal Year", "Fiscal Quarter". Year values: 'FY 2026' (space after FY) Quarter values: 'FY2026 Q2' (no space after FY, one space before Qn) Mappings: - "current/this quarter" → "Fiscal Quarter" = 'FY2026 Q2' - "current/this year" → "Fiscal Year" = 'FY 2026' - "previous/last quarter" → "Fiscal Quarter" = 'FY2026 Q1' - "previous/last year" → "Fiscal Year" = 'FY 2025' If ambiguous, ASK the user.
🧪 Test It — Ask Your Agent:

1. "Revenue for this quarter" → Should filter to FY2026 Q2 (Sep–Nov)

2. "How did we do last year?" → Should filter to FY 2025 (Jun 2024–May 2025)

3. "Revenue for October" → Should map to Q2 and ask about year if ambiguous

10
CASE / Bucket Segmentation
Use Case #9 · Technique: Context Injection
Calculations
Objective

Convert raw numeric values into meaningful business tiers.

Segmentation tiers when user says "classify", "categorize", or "segment": SALES TIERS: Sales < 100: "Low" Sales >= 100 AND < 500: "Medium" Sales >= 500 AND < 1000: "Good" Sales >= 1000 AND < 5000: "Very Good" Sales >= 5000: "Best" Always use non-overlapping ranges. If a value matches no range, label it "Unclassified".
🧪 Test It — Ask Your Agent:

1. "Classify our products by sales performance" → Should apply tiers

2. "What tier is $999 in sales?" → Should answer "Good"

🎉 Intermediate Track Complete!
Your agent now handles calculations, fiscal calendars, hierarchies, and segmentation.
🔴 Advanced Track — Multi-Step, ReAct & Geo Intelligence

Report Shortcuts
Use Case #13 · Technique: Few-Shot
Shortcuts
Objective

Pre-define exactly what common reports contain so the same report comes out consistently.

When user says "Monthly Summary" or "Performance Overview": Generate Table with: - Filter: Current Quarter (FQ Offset = 0) - Metrics: Revenue (Sum), Orders (Count Distinct), Shipping Cost (Sum), Margin % - Group by: Product Category - Sort: Revenue desc When user says "Trend Report": - Revenue + Profit dual-axis Line Chart - Last 4 quarters
🧪 Test It — Ask Your Agent:

1. "Monthly Summary" → Should produce exact table with specified metrics

2. "Trend Report" → Should show dual-axis line chart

3. "Performance Overview for Furniture" → Should add category filter

12
Multi-Step Analysis with Flagging
Use Case #10 · Technique: Prompt Chaining
Calculations
Objective

Teach the agent to perform sequential analysis steps and flag outliers.

For comparative analysis, follow these steps: 1. Identify the comparison dimension 2. Calculate metric for EACH group 3. Compute derived metric (growth rate, variance) 4. Apply threshold/flag if specified 5. Sort to highlight best/worst 6. Present with chart + threshold line "flag", "highlight", "alert" triggers threshold comparison. Show ⚠️ for below target, ✅ for on/above.
🧪 Test It — Ask Your Agent:

1. "Compare revenue growth across regions, flag below 5%" → Should flag underperformers

2. "Which categories had profit decline?" → Should compute change and flag negatives

13
Drill-Down with ReAct
Use Case #14 · Technique: Prompt Chaining
Advanced
Objective

When a user reports an anomaly, the agent investigates step-by-step using the ReAct reasoning pattern.

When user reports an anomaly or asks "why": Apply ReAct drill-down: REASON: What anomaly needs investigation? ACT: Query at current hierarchy level. OBSERVE: Which dimension is the outlier? REASON: Should we drill deeper? ACT: Drill one level into the outlier. REPEAT until root cause found. PRESENT: Summary + supporting chart.
🧪 Test It — Ask Your Agent:

1. "Revenue dropped in West — why?" → Should drill Region→State→City to find root cause

2. "Profit is declining — investigate" → Should drill Category→Sub-Category

14
Period-over-Period Comparison
Use Case #12 · Technique: Context Injection
Time
Objective

Map time expressions to offset values and define comparison formulas.

TIME EXPRESSION MAPPINGS: "this quarter" → FQ Offset = 0 "last quarter" → FQ Offset = -1 "last 4 quarters" → FQ Offset IN (-3, -2, -1, 0) "YTD" → All quarters in current FY up to current FQ Comparison formulas: QoQ % = (Current - Previous) / Previous × 100 YoY % = (Current Year - Previous Year) / Previous Year × 100 Sort time results chronologically.
🧪 Test It — Ask Your Agent:

1. "Revenue trending last 4 quarters?" → Line chart, chronological, with QoQ %

2. "QoQ profit change" → Should compute percentage change

15
Geo Distance & Map Intelligence
Use Case #15 · Technique: Context Injection
Geographic
Objective

Enable location-aware queries using latitude and longitude columns.

Use LATITUDE and LONGITUDE columns for distance calculations. When asked "within X miles of ": 1. Calculate the real-world distance from each record to the specified location. 2. Keep records where Distance <= X miles. 3. Default radius for "near" or "close to": 50 miles. 4. Sort by distance ascending. 5. Default chart: Map.
🧪 Test It — Ask Your Agent:

1. "Stores within 10 miles of downtown Chicago" → Should compute distances, filter, show Map

2. "Customers near Los Angeles" → Should default to 50 miles
